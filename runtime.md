# 泊车系统性能分析平台：从搭建到实战分析

## 1. 项目概述

### 1.1 背景

博世泊车软件（USS7 / Parking）正在从传统 ECU（Renesas/Infineon 单片机）向高阶计算平台（高通 J6E SoC / Linux）迁移。高阶平台上，泊车软件与感知（Horizon）、通信（Viper）、OS 基础设施等多个 Domain 共享 CPU 和内存资源。

关键挑战：
- 传统 ECU 上每个模块独占资源，性能可预测
- 高阶平台上多 Domain 共享资源，存在互相干扰
- 泊车算法有**硬实时约束**——规划超时会导致车辆行为异常
- 需要一个统一的性能度量和分析平台来支撑优化决策

### 1.2 我做了什么

搭建了一套覆盖 **CPU、内存、Runtime** 三个维度的性能分析平台：

| 维度 | 数据来源 | 可视化 | 分析粒度 |
|------|---------|--------|---------|
| **CPU** | vmetrics + 内核采样 | Grafana 时间序列 + 柱状图 | Domain 级 → 进程级 → 函数级 |
| **内存** | /proc/smaps + 进程统计 | Grafana 饼图 + 表格 | Domain 级 → 进程级 → 内存类型细分 |
| **Runtime** | 模块内打点 + rosbag | Grafana 时间序列 + P90/P99 统计 | 模块级 → 场景级 → 函数级（火焰图） |


**怎么读这张表**：

| 内存类型 | 含义 | 高了说明什么 |
|---------|------|------------|
| **stack** | 线程栈空间 | 线程数多，或每个线程栈分配过大 |
| **heap** | 动态分配（malloc/new） | 运行时大量创建对象，可能有内存碎片风险 |
| **bss** | 全局变量（未初始化段） | 全局变量多，或有大型全局数组 |
| **lib** | 共享库（.so 文件） | 依赖的库多或库本身大 |
| **shm** | 共享内存 | 进程间数据传递量大 |
| **binary** | 代码段（.text） | 编译出的代码量大，或包含大型常量表 |
| **anon** | 匿名映射（mmap） | 大块内存分配或内存映射文件 |
| **reserved** | 硬件预留 | 通常是 GPU/DMA buffer，软件无法控制 |

#### 单个 Domain 的饼图

以 Parking 为例：
```
shm:    82,564 KB (34.1%)  ← 共享内存最大，用于模块间传输障碍物地图/轨迹数据
stack:  75,808 KB (31.3%)  ← 线程栈，泊车模块线程数多
bss:    41,276 KB (17.0%)  ← 全局变量区（参数表、状态缓存等）
binary: 30,496 KB (12.6%)  ← 代码段
heap:   11,268 KB (4.7%)   ← 动态分配很少（实时系统尽量避免 malloc）
lib:       830 KB (0.3%)   ← 依赖库很少
```

**分析要点**：实时系统的 heap 应该尽量小（避免运行时动态分配导致延迟抖动），Parking 的 heap 只占 4.7%，说明代码质量好。相反，OS Infra 的 heap 占了 30.9%（61,620 KB），说明它有大量运行时动态操作。

### 2.2 CPU 数据

#### KDMIPS 是什么

KDMIPS = Kilo Dhrystone MIPS（千德莱斯通每秒百万指令数）。它是 CPU 算力的度量单位：
- KDMIPS 越高 = 这个 Domain 消耗的算力越多
- Hist99 = P99 分位的 KDMIPS（99% 的时间内消耗不超过这个值）
- Max = 峰值消耗

```
                  Hist99    Avg      Max     KDMIPS
OS Infra:         20.84    15.57    22.10    ← 最大的 CPU 消费者
Parking:           6.01     2.82     6.76    ← 泊车算法
Horizon:           4.19     1.90     4.85    ← 感知（摄像头/AI）
VRTE:              2.47     1.61     3.20    ← 运行时环境
ViperStageTwo:     1.99     0.83     2.16    ← 通信中间件
```

#### CPU 时间线怎么看

X 轴 = 时间，Y 轴 = CPU 使用率（%），每条线 = 一个 CPU 核心

```
关键模式：
1. 启动尖峰：开头冲到 60-70% → 各模块初始化，正常
2. 稳态运行：降到 40-50% → 正常负载
3. 核心不均衡：某条线持续高于其他线 5-10% → 负载分配问题
4. 周期性毛刺：某条线偶尔突然上升 → 可能是某模块周期性重计算
```

## 3. 分析方法论

### 3.1 四步分析法

```
Step 1: 全局扫描          "哪个 Domain / 进程最吃资源？"
         ↓
Step 2: 异常识别          "有没有不该高的指标偏高？Max/Avg 比值是否异常？"
         ↓
Step 3: 根因定位          "为什么高？是算法问题？调度问题？平台差异？"
         ↓
Step 4: 量化验证          "优化后指标降了多少？功能是否无损？"
```

### 3.2 关键指标和判断标准

| 指标 | 健康范围 | 警告 | 危险 |
|------|---------|------|------|
| 进程 CPU P90 | < 50% 调度周期 | 50-70% | > 70% |
| 进程 CPU P99 | < 70% 调度周期 | 70-85% | > 85% |
| P99/P90 比值 | < 1.2 | 1.2-1.5 | > 1.5（有严重尖峰） |
| Max/Avg 比值 | < 2.0 | 2.0-3.0 | > 3.0（波动剧烈） |
| Domain 内存增长 | 稳定 | 缓慢增长 | 持续增长（内存泄漏） |

### 3.3 常见性能问题模式

| 模式 | 特征 | 原因 | 对应案例 |
|------|------|------|---------|
| **持续高位** | P90 和 P99 都很高且接近 | 算法本身计算量大 | NFM Hull 算法 |
| **尖峰型** | P99 远高于 P90 | 特定场景/阶段触发重计算 | Planner 最后一把 |
| **平台差异型** | 同代码在不同平台表现差异大 | 平台特性不同（MCU vs Linux） | OS Infra 轮询 |
| **通信瓶颈型** | 通信相关进程 CPU 高 | 序列化/IPC 开销 | io-socket 异常 |
| **阶梯上升型** | 随版本迭代逐步升高 | 新功能累加未同步优化 | PMP 资源逐步上升 |

## 4. 实战案例

### 案例 1：NFM Runtime 优化的全程追踪

#### 4.1.1 发现问题

在进程级 P90/P99 表中，NFM 占据了绝对第一位：

```
/asw/RbpNfm_activity:  Cpu P90 = 61.9%,  Cpu P99 = 62.2%
```

NFM 的调度周期是 40ms。P90 = 61.9% 意味着 90% 的情况下 NFM 执行耗时约 24.8ms，只剩 15.2ms 的余量。如果后续增加功能，很可能超时。

同时 P99/P90 = 62.2/61.9 = 1.005，说明 NFM **几乎每个周期都很慢**——不是偶尔尖峰，而是**算法本身计算量就大**。这指向算法级优化而非调度优化。

#### 4.1.2 定位根因

团队通过火焰图和代码分析，定位到 NFM 内部的 **Hull（轮廓提取）算法**是主要耗时来源。Hull 算法从超声波传感器的散点数据中提取障碍物外轮廓多边形，计算复杂度随点数增长。

#### 4.1.3 分阶段优化与验证

这是最有价值的数据——通过平台追踪了整个优化过程的每一步效果：

| 时间 | 软件版本 | P90 | P99 | 变化 | 说明 |
|------|---------|-----|-----|------|------|
| 04/10 | daily dev（优化前基线） | **22.9%** | **23.7%** | — | 基线测量 |
| 04/10 上午 | dev + PR1 替换 | 15.5% | 21.1% | P90↓32% | 第一批 Hull 优化合入 |
| 04/10 下午 | dev + PR1 替换（更长测试） | 11.8% | 17.6% | P90↓48% | 延长测试验证稳定性 |
| 04/13 | dev + PR1/2/3 及依赖替换 | 12.8% | 16.5% | P99↓30% | 第二三批优化合入 |
| 04/16 上午 | dev + PR1/2/3 及依赖替换 | 11.8% | 11.9% | **P99↓50%** | P99 大幅下降！ |
| 04/16 下午 | 同上（不同时段验证） | 12.7% | 15.9% | 略回升 | 下午场景更复杂 |
| **04/20** | **dev + 13 个 PR 全部合入** | **11.8%** | **14.5%** | **P90↓48%, P99↓39%** | **最终结果** |

注意：这里的 P90/P99 百分比是 `RbpEnvPer_activity`（包含 NFM 的环境感知进程）的 CPU 占用百分比。

#### 4.1.4 分析要点

**为什么 04/10 上午和下午结果不同**（P90 从 15.5% 降到 11.8%）？

同一个软件版本（dev + PR1），但下午测试时间更长（2 小时 vs 3 小时），样本量更大，统计结果更稳定。这说明**测试时长对 P90/P99 有影响，短时测试可能高估或低估**。

**为什么 04/16 上午 P99=11.9% 但下午 P99=15.9%**？

上午测试以简单场景为主（直行、标准车位），下午测试包含更多困难场景（斜车位、狭窄空间）。P99 对尾部场景敏感，困难场景下 NFM 处理的点云更密集，Hull 算法耗时更长。

**最终效果**：

```
优化前（04/10 基线）:  P90 = 22.9%,  P99 = 23.7%
优化后（04/20 全量）:  P90 = 11.8%,  P99 = 14.5%

P90 下降: 22.9% → 11.8%  (↓ 48.5%)
P99 下降: 23.7% → 14.5%  (↓ 38.8%)

对应 OPL 结论："Hull 的优化导入后 NFM 的整个 runtime 优化已经达到 50%"
```

### 案例 2：OS Infra 异常 CPU 占用

#### 4.2.1 发现问题

在 Domain 级 CPU 使用图表中，发现了一个反直觉的结果：

```
                  Hist99    Avg      Max     KDMIPS
OS Infra:         20.84    15.57    22.10    ← 第 1 名！
Parking:           6.01     2.82     6.76    ← 只排第 2
```

OS Infra 是操作系统基础设施层，按常理不应该比泊车算法还耗 CPU。它的 Hist99 是 Parking 的 **3.5 倍**。

#### 4.2.2 辅助线索：内存构成

从内存数据进一步分析 OS Infra 的行为特征：

```
OS Infra 内存构成:
  heap:   61,620 KB (30.9%)  ← 动态分配占比最高，说明有大量运行时操作
  anon:   42,484 KB (21.3%)  ← 匿名映射高，可能是大块数据的内存映射
  stack:  36,508 KB (18.3%)  ← 线程数不少
  binary: 31,184 KB (15.6%)  ← 代码量大

对比 Parking:
  heap:   11,268 KB (4.7%)   ← 动态分配很少（实时系统规范）
  shm:    82,564 KB (34.1%)  ← 主要用共享内存传递数据
```

OS Infra 的 heap 是 Parking 的 **5.5 倍**，说明 OS Infra 有大量的运行时动态内存操作。在 Linux 上，频繁的 malloc/free 不仅消耗 CPU，还可能触发缺页异常。

#### 4.2.3 根因定位

结合代码审查，定位到 `rbp_infra` 中的 **config master 模块以 10ms 为周期轮询配置参数**。

在传统 ECU 上，config master 直接读取 Flash 寄存器，耗时 ~1 个时钟周期。但在 Linux SoC 上，每次轮询涉及：
1. 进程间通信（IPC）
2. 内存拷贝和序列化
3. 可能的上下文切换

10ms 一次意味着每秒 100 次这样的重操作。

#### 4.2.4 优化方案与结果

将 config master 轮询周期从 **10ms 降到 1000ms**。

合理性分析：config master 读取的是配置参数（类似 ParGen 生成的那种参数），参数值在运行过程中极少变化（只有诊断工具在线修改时才会变），1 秒检查一次完全足够。

```
优化结果：
  OS Infra CPU 消耗显著下降
  "目前 infra runtime 结果看符合预期，此改动平台也是 1 秒，无风险"
  释放的 CPU 余量可用于后续功能集成（行泊一体）
```

#### 4.2.5 关键思考：平台差异型问题

这个案例的本质是**同一段代码在不同平台上的性能特征完全不同**：

```
传统 ECU（MCU）:
  config master 每 10ms 读一次 Flash 寄存器
  耗时: ~100ns（纳秒级，可忽略）

高阶平台（Linux SoC）:
  config master 每 10ms 做一次 IPC + memcpy + 序列化
  耗时: ~1ms（占 10ms 周期的 10%）
  × 100 次/秒 = 大量 CPU 浪费
```

这类问题不会在代码 review 中被发现，只有通过性能平台的**数据对比**才能暴露出来。

### 案例 3：Parking Domain 尖峰分析——"最后一把" 问题

#### 4.3.1 发现问题

在 Domain 级数据中，Parking 的 Max/Avg 比值异常高：

```
Parking:  Avg = 2.82 KDMIPS,  Max = 6.76 KDMIPS
          Max / Avg = 2.4 倍

对比其他 Domain:
  OS Infra:  Max/Avg = 22.10/15.57 = 1.42 倍  ← 相对稳定
  Horizon:   Max/Avg = 4.85/1.90  = 2.55 倍   ← 也有尖峰
  VRTE:      Max/Avg = 3.20/1.61  = 1.99 倍   ← 比较稳定
```

Parking 的 Max/Avg = 2.4 意味着：大部分时候泊车模块很轻松（2.82 KDMIPS），但偶尔 CPU 消耗会飙升到 6.76——几乎是平时的 2.5 倍。

#### 4.3.2 场景关联

结合进程级数据和测试场景日志，发现尖峰全部出现在**泊车最后一把**：

```
泊车过程中各阶段的 CPU 消耗（示意）:

  搜索车位    第1把    第2把     最后一把
  ─────────  ──────  ──────   ──────────
  PSD 活跃   Planner  Planner  Planner
  NFM 低负载  单次规划  单次规划  实时重规划 × N
  Planner 空闲                  碰撞检测密集
  
  CPU:  低     中      中       高（尖峰！）
```

为什么"最后一把"特殊：
1. **空间极其紧凑**：车辆已经接近最终位置，与障碍物间距小
2. **需要反复重规划**：每次微调方向后都要重新检查是否会碰撞
3. **碰撞检测更精细**：距离近，需要检查更多点和更多障碍物
4. **多次连接尝试**：路径规划的 connection 算法要尝试多种路径

#### 4.3.3 具体进程数据印证

从进程级 P90/P99 数据：

```
/asw/RbpPmpPlanning:  P90 = 36.3%,  P99 = 41.3%
                      P99/P90 = 1.14  ← 接近持续高位模式

/asw/RbpPmpMain_activity:  P90 = 17.1%,  P99 = 19.3%
                           P99/P90 = 1.13

两个 PMP 进程合计:  P90 ≈ 53.4%,  P99 ≈ 60.6%
```

PMP + Planner 两个进程合计占了 **超过 60% 的 CPU 周期**（P99），仅次于 NFM。

#### 4.3.4 优化方向

基于以上分析，Planner 团队识别出多个优化方向：

| # | 优化方向 | 原理 | 预期收益 | 状态 |
|---|---------|------|---------|------|
| 1 | 减少实时规划调用频率 | 到终点偏差阈值作为参数，允许更大的位置偏差 | 减少 Planner 调用次数 | 已提交 |
| 2 | 减少单次规划的 loop 次数 | 限制每次规划尝试的连接次数 | 减少单次耗时 | 已提交 |
| 3 | RS 曲线长度预检 | 在碰撞检测前先检查路径长度是否满足最小起步距离 | 提前排除不可行路径 | 已合入 |
| 4 | 碰撞检测矩形简化 | 4 个轮子轮廓 → 1 个矩形 | 减少碰撞检测计算量 | 测试中 |
| 5 | 减少数据拷贝 | 值传递 → 引用传递 | 减少 memcpy 开销 | AI Agent 分析后提出 |
| 6 | 循环外提 | 把循环内不变的计算提到循环外 | 减少冗余计算 | 分析中 |

### 案例 4：io-socket 通信异常

#### 4.4.1 发现问题

在进程级数据中，`/system/io-sock` 出现在 CPU Top 5：

```
/system/io-sock:  P90 = 20.2%,  P99 = 20.4%
```

io-socket 是底层通信进程，不包含业务逻辑，不应该占这么高的 CPU。而且 P99/P90 = 1.01，说明它**持续高位**而非偶尔尖峰。

#### 4.4.2 分析

结合 OPL 记录：

> "io-socket 的 runtime 也有异常，跟录数据/someip 有关"

SOME/IP（Scalable service-Oriented Middleware over IP）是车载以太网的通信协议。io-socket 负责底层的 socket 收发。

异常原因：测试车上同时在录制 rosbag 数据，导致 SOME/IP 通信量翻倍——每条消息不仅发给业务模块，还要发给录制工具。这相当于 io-socket 的负载人为翻倍了。

#### 4.4.3 结论

```
判断：io-socket 的高 CPU 是测试环境的副作用，非量产代码问题
      - 量产环境不会有 rosbag 录制
      - "从业务代码角度 ASW 无法在量产软件里做改善，需要 SOME/IP 考量"
      
建议：
      - 录制数据时的 runtime 测量需要打标记，区分"有录制"和"无录制"
      - 后续行泊一体集成时再统一评估通信层优化
```

这个案例展示了分析中一个重要技能：**区分真问题和环境干扰**。

### 案例 5：MOT 降频的 trade-off 分析

#### 4.5.1 背景

MOT（Motion Object Tracking，运动目标跟踪）负责跟踪其他运动物体（行人、其他车辆）的位置和速度。

#### 4.5.2 优化方案

将 MOT 的调度周期从 **10ms 改为 20ms**。

```
优化前: 每秒执行 100 次，CPU P90 = X%
优化后: 每秒执行  50 次，CPU P90 ≈ X/2（直接减半）

实际测量："20ms 情况下 P90 相比于 10ms 下降了 50%"
```

#### 4.5.3 trade-off 分析

降频是最简单粗暴的优化——效果立竿见影，但有功能代价：

| 指标 | 10ms | 20ms | 影响 |
|------|------|------|------|
| CPU 占用 | 100% | 50% | 大幅降低 |
| 跟踪延迟 | 10ms | 20ms | 增加 10ms |
| 位置精度 | 更高 | 略低 | 快速运动场景可能受影响 |
| 泊车场景影响 | — | — | 泊车低速（<10km/h），20ms 内物体移动 <5.6cm，影响可忽略 |

#### 4.5.4 结论

```
"已经在极氪测试过，功能正常，测试结果上看性能上没有什么损失"
"mot 暂时不导入，视最终系统级优化效果评估是否导入到量产"

分析：MOT 降频在泊车场景下功能无损（低速），但团队采取保守策略，
先观察其他模块优化后的系统级效果，再决定是否需要 MOT 降频这张"底牌"。
```

### 案例 6：内存维度的系统画像

#### 4.6.1 发现

从内存总表中识别出几个关键特征：

**ION 的 1GB reserved**：
```
ION: 1,048,576 KB = 1GB，100% 是 reserved
```
这是 SoC 的 GPU/DMA 硬件预留缓冲区，软件无法控制也不需要优化。分析时应**排除 ION**，实际软件使用内存为 2383 - 1024 = **1.36 GB**。

**Parking 的 shm 占比**：
```
Parking shm = 82,564 KB (34.1%)，约 80MB
```
这是泊车各模块通过共享内存传递数据：NFM 把障碍物地图写入 shm → PMP/Planner 从 shm 读取。80MB 的 shm 说明**模块间数据传递量大**，如果能减少传递的数据量（比如 NFM 输出更精简的轮廓），shm 占用也会下降。

**MTA socket gateway 的内存异常**：
```
/asw/bin/.mta_socket_gateway:  Mem P99 = 335 MiB
```
一个网关进程占了 335MB 内存，这在嵌入式系统中非常罕见。可能存在缓冲区过大或内存泄漏，需要进一步排查。

## 5. 分析技巧总结

### 5.1 看数据的顺序

```
1. 先看 Domain 级总览  → 哪个 Domain 最大？占比合理吗？
2. 再看进程级排名      → 该 Domain 内哪个进程最高？
3. 然后看时间趋势      → 是持续高还是偶尔尖峰？在什么阶段/场景高？
4. 最后看代码级        → 火焰图找热点函数，分析为什么慢
```

### 5.2 有用的比值

| 比值 | 计算方式 | 意义 |
|------|---------|------|
| P99/P90 | P99 ÷ P90 | 接近 1.0 = 稳定高位；> 1.5 = 有严重尖峰 |
| Max/Avg | Max ÷ Avg | > 2.0 说明有极端情况，需要关注尾部场景 |
| Gap Cpu P99 | 优化前 P99 - 优化后 P99 | 优化效果的直接度量 |
| Gap Mem P99 | 同上但看内存 | 负值 = 内存优化了；正值 = 内存增加了 |

### 5.3 常见陷阱

| 陷阱 | 说明 | 应对 |
|------|------|------|
| **测试环境干扰** | 录制 rosbag 会增加通信负载 | 区分有录制/无录制的数据 |
| **测试时长不足** | 短时测试可能高估或低估 P90/P99 | 至少测试 2 小时，覆盖多种场景 |
| **场景不充分** | 只测简单场景会低估 P99 | 必须包含困难场景（斜车位、狭窄空间） |
| **不同车辆差异** | 不同测试车（zeekr 15652C vs 17388C）硬件状态不同 | 对比时尽量用同一辆车 |
| **时段差异** | 上午/下午的测试场景分布不同 | 记录每次测试的场景类型 |

## 7. 技术实现

### 7.1 数据采集

| 数据类型 | 采集方式 | 频率 |
|---------|---------|------|
| CPU 使用率 | vmetrics（Kubernetes 监控） | 每秒 |
| 内存使用 | /proc/[pid]/smaps 解析 | 每 5 秒 |
| Runtime 打点 | 模块内时间戳 → rosbag | 每调度周期 |
| 火焰图 | perf record + flamegraph.pl | 按需采集 |

### 7.2 数据处理脚本

```
采集数据 (rosbag / vmetrics export)
    │
    ▼
Python 解析脚本
    ├── 解析 smaps → 按 Domain/进程/内存类型 聚合
    ├── 解析 runtime 打点 → 计算 P90/P99/Max/Avg
    ├── 解析 perf 数据 → 生成火焰图 SVG
    └── 推送到 Grafana 数据源 (VictoriaMetrics / InfluxDB)
    │
    ▼
Grafana Dashboard
    ├── Overview: Domain 级 CPU + 内存总览
    ├── Memory: 各 Domain 内存细分饼图 + 表格
    ├── CPU: 时间序列 + Domain 对比柱状图
    ├── Process: 进程级 P90/P99 排名表
    └── Runtime: 模块级 runtime 时间序列 + 场景分析
```

### 7.3 关键技术点

| 技术点 | 说明 |
|-------|------|
| P90/P99 计算 | 对采集的 runtime 样本排序，取第 90/99 百分位 |
| 内存分类 | 解析 /proc/smaps 的 VmRSS、Shared、Private 等字段 |
| 火焰图生成 | perf record → perf script → stackcollapse-perf.pl → flamegraph.pl |
| Gap 计算 | 同一进程不同版本/时间段的 P99 差值，用于量化优化效果 |
| 数据对齐 | rosbag 时间戳与 vmetrics 时间戳对齐，实现场景-资源关联分析 |
